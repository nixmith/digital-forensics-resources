Introduction

The Sleuth Kit (TSK) is an open-source collection of command-line tools for forensic analysis of disk images and filesystems. In this tutorial you will build a practice image with two partitions and known content, then work through TSK commands layer by layer — from volume analysis down to individual blocks — verifying each tool’s output against raw hex. Later tasks chain tool outputs together and build toward realistic forensic exercises.

Every TSK command follows a consistent naming convention: the prefix tells you the layer and the suffix tells you the operation.

TSK Naming Convention

The Sleuth Kit organizes its tools into layers. Each prefix maps to a forensic abstraction — from the raw image container down to individual data blocks.

The suffix tells you the operation: stat = show metadata, ls = list entries, cat = extract content, find = search or resolve names.

For example, mmls is media management + list — it lists partitions. icat is inode + cat — it extracts file content by inode number.

Nearly every command after img_stat takes an -o flag specifying the sector offset to the partition. You will see this pattern repeatedly.

Prefix Layer Operations
img Image file img_stat
mm Media management (partitions) mmls, mmcat
fs Filesystem fsstat, fls
i Inode / metadata istat, icat, ifind
blk Data unit (block) blkstat, blkcat, blkls

Task 0: Build the Practice Image

The setup script creates a 128 MiB raw disk image with two ext3 partitions. It populates partition 1 with files in three directories, copies some to partition 2 as a “USB backup,” then deletes four files to create recovery targets. Download the script directly.

#!/usr/bin/env bash
# setup_ext3_lab.sh — Create a 128M forensic practice image (two partitions)
set -euo pipefail

IMG="ext3_lab.dd"
MNT1="/mnt/ext3_lab_p1"
MNT2="/mnt/ext3_lab_p2"

dd if=/dev/zero of="$IMG" bs=1M count=128 status=none

sfdisk --quiet "$IMG" <<EOF
label: dos
start=2048, size=129024, type=83
start=131072, type=83
EOF

LOOP1=$(losetup --find --show --offset $((2048*512)) --sizelimit $((129024*512)) "$IMG")
LOOP2=$(losetup --find --show --offset $((131072*512)) "$IMG")
mkfs.ext3 -L "evidence_vol" -q "$LOOP1"
mkfs.ext3 -L "usb_backup"   -q "$LOOP2"

mkdir -p "$MNT1" "$MNT2"
mount "$LOOP1" "$MNT1"
mount "$LOOP2" "$MNT2"

# Phase 1-3: create files with sleep gaps for timeline separation
echo "The quick brown fox jumps over the lazy dog." > "$MNT1/readme.txt"
mkdir "$MNT1/documents"
echo "SSN: 078-05-1120 (original SS-5 sample)" > "$MNT1/documents/pii_sample.txt"
echo "Contract value: \$4,500,000. Effective 2025-03-01." > "$MNT1/documents/contract.txt"
sync; sleep 2

mkdir "$MNT1/projects"
echo "Project ATLAS: Phase 2 approved. Budget \$1.2M. Lead: J. Chen." \
  > "$MNT1/projects/atlas_status.txt"
echo "Meeting 06/15: merger timeline with legal. NDA expires July 1." \
  > "$MNT1/projects/meeting_notes.txt"
sync; sleep 2

mkdir "$MNT1/personal"
echo "Flight: LAX→JFK Mar 15 2025, seat 12A, conf# BK7942" > "$MNT1/personal/travel.txt"
echo "Password reminder: bank pin 4421, backup code XKCD-9371" > "$MNT1/personal/passwords.txt"
echo "Reminder: pick up prescription Tuesday" > "$MNT1/personal/todo.txt"
sync; sleep 2

# Phase 4: copy to "USB backup" partition, leave a transfer log
cp "$MNT1/documents/contract.txt"       "$MNT2/contract_backup.txt"
cp "$MNT1/projects/atlas_status.txt"    "$MNT2/atlas_backup.txt"
echo "Backup $(date -u +%Y-%m-%d): contract.txt, atlas_status.txt" \
  > "$MNT2/backup_manifest.txt"
echo "Sensitive files copied to USB backup (partition 2)." \
  > "$MNT1/documents/transfer_log.txt"
sync; sleep 2

# Phase 5: create one more file then delete four
echo "SECRET: The meeting is at midnight, east dock." > "$MNT1/deleted_note.txt"
sync; sleep 1

# (script prints inode/md5 table for each file before deleting)
rm "$MNT1/deleted_note.txt" "$MNT1/personal/passwords.txt"
rm "$MNT1/projects/atlas_status.txt" "$MNT1/projects/meeting_notes.txt"
sync

umount "$MNT1"; umount "$MNT2"
losetup -d "$LOOP1"; losetup -d "$LOOP2"
rmdir "$MNT1" "$MNT2"

Run with sudo bash setup_ext3_lab.sh. The script prints a reference table with inode numbers and MD5 hashes for the four deleted files — save this output, you will need it for Tasks 9 and 11.

Task 1: Examine the Disk Image (img_stat)

Before diving into partitions, confirm the image basics. img_stat reports the container format and total size.

The image is raw (byte-for-byte copy) and 134,217,728 bytes — exactly 128 MiB. TSK also supports EWF (E01), AFF, and other forensic formats; use -i raw to force the type if auto-detection fails.

This size matters: 134,217,728 / 512 = 262,144 total sectors. Any partition boundary beyond that would indicate corruption.

Task 2: Identify Partitions (mmls)

mmls reads the partition table from the first sector of the image.

This image has two partitions. Row 002 starts at sector 2048 — this is the main evidence volume. Row 003 starts at sector 131072 — this is the USB backup partition.

The Start column gives you the -o offset for all subsequent commands. For partition 1: -o 2048. For partition 2: -o 131072. We will focus on partition 1 through Task 8, then use both in Task 13.

The type code 0x83 means “Linux native.” Let’s verify partition 1 against the raw MBR.

xxd dumps raw bytes from the image. The -s 446 seeks to byte 446 where MBR partition entries begin, and -l 32 reads 32 bytes — enough for both 16-byte entries.

The first entry’s type byte (offset 4) reads 0x83 — Linux, confirming mmls row 002. Its LBA start at bytes 8–11 reads 00 08 00 00 little-endian = 2048. The second entry follows the same layout at the next 16 bytes, also type 0x83, with LBA start 00 00 02 00 = 131072. Both match mmls exactly.

Task 3: Filesystem Details (fsstat)

With the partition offset known, fsstat reveals ext3 internals.

The volume name evidence_vol matches our label — useful for correlating with mount logs. The timestamps (last written, checked, mounted) establish system activity independent of file timestamps.

Inode 2 is always the root directory on ext filesystems. The block size of 4096 bytes is critical for byte-offset math: a file in block N lives at absolute offset (2048 × 512) + (N × 4096) in the image.

Cross-check the block size against the raw superblock. The ext3 superblock starts at byte 1024 within the partition — absolute offset (2048 × 512) + 1024 = 1,049,600. The xxd command below reads 64 bytes starting there.

The superblock encodes block size as a log2 shift. Value 0x02 means 2^(2+10) = 4096 bytes — matching fsstat.

Task 5: Examine Inode Metadata (istat)

istat dumps all metadata for a given inode. Inspect readme.txt using the inode from fls (inode 12).

The file is Allocated, 46 bytes, 1 hard link. The three ext3 timestamps — access, modification, inode change — are the MAC times used in timeline analysis. ext3 does not store a creation (birth) time; that was added in ext4.

Direct Blocks: 1094 tells you which filesystem block holds the data. This is the link between istat output and icat/blkcat input — you will use this block number in Task 6. Larger files would show multiple direct blocks and indirect block pointers.

Now inspect a deleted file’s inode. Use the inode from fls for deleted_note.txt:

Not Allocated with link count 0 — unlinked from the directory. But size, timestamps, and block pointers are all intact. This is what makes recovery possible.

The Inode Modified time is one second later than File Modified — that is the deletion event. rm updated the link count (inode metadata) without touching file content. The block number here (1095) becomes the input to blkcat in Task 6 and blkstat to check allocation status.

Task 7: Search Unallocated Space (blkls)

When you do not know which inode held a deleted file — or the inode has been wiped — search the raw unallocated blocks. blkls extracts every unallocated block into a binary blob that you can search with strings and grep.

Both deleted messages appear. This works even without knowing the inode — blkls bypasses the metadata layer entirely and dumps raw block content. Tools like bulk_extractor extend this approach with pattern matching for emails, URLs, credit card numbers, and other structured data.

Try it: Search for “ATLAS” in unallocated space. Now search for “contract” — does allocated file content appear in unallocated space? Consider what it would mean if it did.

$ strings unallocated.bin | grep -i "secret"
